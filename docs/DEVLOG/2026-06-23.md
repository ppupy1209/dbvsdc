# 2026-06-23 — 프로젝트 시작: 기획과 인프라 결정

> 개발기 첫 글. 나중에 yeonwoo.dev 스타일로 발행 예정.

## 무엇을 만드나

퇴직연금 DB형 vs DC형을 비교·시뮬레이션하는 사이트. 핵심 질문은 단순하다:
**"내 조건에서 수익률 몇 %부터 DC가 DB를 이기는가?"**

## 오늘의 고민: 어디에 배포할 것인가

제약이 까다로웠다. (1) 저비용, 가능하면 무료. (2) 백엔드는 반드시 Java/Spring. (3) DB 쓰면 MySQL.

### 문제 1 — "AWS에 인스턴스 2대 띄우면 무료 아닌가?"

아니었다. EC2 프리티어 750시간은 **인스턴스당이 아니라 계정 합산**이다. 2대를 24시간 돌리면 1,500시간 → 초과분 과금. 게다가 2025-07-15부터 신규 계정은 12개월 무료가 아니라 **크레딧 소진형**으로 바뀌었다.

### 문제 2 — "그럼 프리티어 계정을 하나 더 만들면?"

기각. AWS 약관 위반이고, 카드·IP·디바이스로 동일인이 연결되면 **chipthrone가 올라간 기존 계정까지 정지될 수 있다.** 운영 중인 서비스를 걸 도박이 아니다.

### 문제 3 — Java/Spring은 Vercel serverless에 안 맞는다

처음엔 "프론트+백 전부 Vercel serverless"를 떠올렸지만, JVM은 콜드스타트가 무거워 부적합. Spring은 상시 켜진 서버 1대가 필요하다.

### 결론 — 프론트 Vercel, 백엔드 NCP

영구 무료만 보면 Oracle Always Free(MySQL HeatWave 무료 포함)가 최선이었지만,
**타깃이 한국 퇴직연금 사용자**라는 점 때문에 NCP를 택했다. 국내 지연시간, 원화 결제, 한국어 운영, 금융/공공 친화. 크레딧 3개월 후 소액 과금은 감수.

상세 결정: [ADR 0001](../DECISIONS/0001-infra-ncp.md)

## 배운 것 / 설계 포인트

- "예상 수익률"을 주는 API는 없다. 과거 실적만 받아올 수 있고, 미래는 사용자가 슬라이더로 가정하게 하는 게 정확성·법적 리스크 모두에서 낫다.
- 공공데이터 API는 느리고 제한이 있어 일배치 캐싱이 정석.

## 오후: 프론트 디자인 시안 반복 (고급·심플)

핵심 화면(시뮬레이터)을 시안으로 여러 번 돌렸다. 컨셉은 평면 흰 카드 + 얇은 보더 + 색 2개(DB 무채색 / DC 블루).

확정한 기능:
- **연도별 그래프**: DB vs DC 적립금 추이, DC 역전 시점 표시.
- **30/70 백테스트**: 안전자산 30%(예금) + 위험자산 70%(S&P500·나스닥·다우·코스피·코스닥). 30/70은 우연이 아니라 **DC 법정 위험자산 한도**와 일치 → 현실적 최대 공격형 기준선.
- **두 모드**: 과거 백테스트(실제 경로, 하락장 출렁임) / 미래 예측(과거 평균수익률 투영).
- **기간 1~30년** 슬라이더(데이터 1995~2024로 확장).
- **세금**: 퇴직소득세 산식(근속연수공제·환산급여·누진세율) 구현. **IRP 연금수령 시 30% 감면** 표시.
- **안전자산 금리 2026년 기준 3.0% 고정**.
- **정확도 안내 문구** UI에 명시(예시값·환율·리밸런싱·세금추정·미래예측 한계).

데이터 소스 결론:
- "한 사업자 DC 라인업+수익률"은 **통합연금포털 Open API**가 적합(승인 필요).
- ETF 시세는 KRX/증권상품시세정보, 또는 KIS Developers(무료, 계좌 필요).
- 지수는 **국내 추종 ETF**로 구현하면 환율·보수 자동 반영(권장).

미해결(정밀화 TODO): 환율 반영, DC 운용수익 과세 분리, 법정 산식 근거 확인, 세율·금리 연 갱신.

## 저녁: 시안 → 실제 코드 (Next.js)

`web/`에 Next.js 16 + React 19 + Tailwind v4로 프론트 MVP를 스캐폴딩하고 시뮬레이터를 이식했다.

- Next 16은 breaking change 안내가 있었지만(번들 docs 확인), 표준 App Router 패턴(metadata export, client component)이라 그대로 적용.
- 계산 로직은 `src/lib/calc.ts`에 **순수 함수**로 분리 — 테스트하기 쉽고, 나중에 백엔드(Java)로 로직을 옮길 때 기준이 된다.
- 한글 폰트 위해 next/font(라틴) 대신 시스템 폰트 스택 사용.
- `npm run build` 통과, 로컬 dev 서버 렌더 확인.

문제: 로컬 포트 3000은 chipthrone 등 다른 node 프로세스가 점유 중 → preview를 autoPort로 우회.

## 밤: 랜딩 섹션 + 계산식 정밀화

랜딩(Intro) 추가 — "DB로 둘까, DC로 굴릴까?" 히어로 + DB/DC 30초 요약 카드 + 시뮬레이터로 스크롤 CTA.

계산식 정밀화(법정 근거 확인):
- DB = 30일분 **평균임금** × 근속(상여·수당 포함). 모델은 연봉/12로 근사.
- **임금피크제 옵션** 추가: DB는 최종 평균임금 기준이라 임금피크 시 직접 감액 → DC와의 구조적 차이를 토글로 체험. 이게 사람들이 임금피크 직전 DC로 갈아타는 이유.
- 세금 과세범위 정정: DC 회사부담금+운용수익 전액이 퇴직소득세 대상(현행 일치). 개인 추가납입분만 별도(기타/연금소득세) — DB vs DC 비교에선 범위 밖.

빌드 통과, 브라우저 렌더 확인(랜딩+토글). 첫 스크린샷이 빈 화면이었는데 재컴파일 타이밍 문제였고 데스크톱 뷰포트에서 정상.

## 심야: 실데이터 연동 준비 (데이터 주도 전환)

"실제 지수/ETF 데이터" 연동을 진행. 단 실데이터는 API 키(승인 대기)+Spring 서버가 있어야 해서, 지금은 **연동 가능한 상태로 만드는** 데 집중.

- 계산 로직을 데이터 주도로 리팩터링: `calc.simulate(...)`가 `MarketData`를 인자로 받음(예시값 하드코딩 제거).
- `lib/api.ts`: `fetchMarketData()` — `NEXT_PUBLIC_API_BASE`의 백엔드 호출, 실패 시 예시 데이터 폴백 → UI는 항상 동작.
- mock 라우트 `app/api/index-returns/route.ts`: 계약 형태로 예시 데이터 응답(개발용, Spring이 대체).
- UI에 **데이터 소스 배지**("예시 데이터" ↔ "실데이터·기준일") 추가.
- 백엔드 계약·MySQL 스키마를 [api-spec.md](../api-spec.md)에 정의 → Codex 위임 준비 완료.

핵심 설계: 백엔드가 준비되면 **`NEXT_PUBLIC_API_BASE` 한 줄**만 바꾸면 실데이터로 전환. 프론트 코드 수정 불필요.

검증: build 통과, mock 엔드포인트 curl 응답 확인(계약 일치), 콘솔 에러 없음. (스크린샷 도구는 이 세션 내내 불안정.)

## 백엔드 위임: Codex가 Spring 스캐폴딩

api-spec.md를 기준으로 Codex(codex-rescue)에 백엔드를 위임 → `api/`에 Spring Boot 3.3.7/Java 17 프로젝트 생성.

- GET /api/index-returns 샘플모드 구현(프론트 indexData.ts 값과 일치).
- `MarketDataSource` 인터페이스 + `SampleMarketDataSource`(기본) / `LiveMarketDataSource`(TODO 스텁).
- 프로필 분리: 기본=sample(DB 없이 부팅, DataSource/JPA/Flyway autoconfig 제외) / `live`=MySQL.
- MySQL 스키마(Flyway V1), JPA 엔티티/레포, @Scheduled 배치 골격(live only), CORS, env 시크릿.
- Claude 리뷰: 계약·샘플값·프로필·CORS·시크릿·스키마 전부 확인 통과.

한계: 이 PC는 Java 8라 빌드 불가 → 코드 리뷰로 검증. 빌드·실행은 JDK 17(NCP 또는 로컬 설치) 필요. 실데이터는 키+LiveMarketDataSource 구현 후.

## Docker 로컬 실행 환경

로컬 Java가 8이라 Spring Boot 3을 못 돌리는 문제를 **Docker로 우회**.

- `api/Dockerfile`: 멀티스테이지(maven:3.9-temurin-17 빌드 → temurin-17-jre 실행).
- `docker-compose.yml`: api(8090) + mysql(3307). api 기본 `sample` 프로필이라 DB 없이도 동작, mysql은 live 단계용.
- 결과: **이미지 빌드 성공**(= Codex 코드가 JDK17에서 실제 컴파일됨, 그동안 리뷰로만 검증했던 것 실증) → `docker compose up` → `GET /api/index-returns` **HTTP 200**, 계약대로 JSON(sample, 1995~2024, 값 일치) 확인.

이제 로컬에서 풀스택 기동 가능. JDK 설치 불필요.

## 프론트↔백엔드 연결 검증 (end-to-end)

`web/.env.local`에 `NEXT_PUBLIC_API_BASE=http://localhost:8090` 설정 후 dev 서버 재시작.
- 네트워크: 프론트(localhost:3000)가 `GET http://localhost:8090/api/index-returns` → **200, CORS 에러 없음**.
- 응답 = Spring 백엔드의 계약 JSON(sample). 프론트가 소비해 배지 "예시 데이터" 표시, 시뮬레이터 정상 렌더.
- `.env.local`은 gitignore. 다른 PC용으로 `web/.env.example` 추가.

전체 데이터 흐름(프론트 → 백엔드 → 화면)이 한 줄로 연결됨을 실증. 백엔드만 `LiveMarketDataSource`로 채우면 실데이터로 전환.
